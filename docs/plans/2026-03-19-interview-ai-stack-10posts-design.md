# 2026-03-19 面试博客：10 篇独立小项目设计

## 背景与目标

招聘要求核心点包括：

1. 前端（React/更好是 Next.js）：大模型 API 调用、前端 Streaming（类似打字机效果）、复杂对话上下文状态管理。
2. 后端底子（Go 或 Rust）：完成可运行项目、干净业务逻辑、能排错并完成编译/运行闭环。
3. AI 工具驱动（重点）：Claude Code / GitHub Copilot / Cursor 重度使用，用 AI 快速写掉基础 CRUD 与样板代码，把精力投入核心逻辑。
4. 独立部署：在普通 Linux 服务器上用 Docker、Nginx 反代、证书、跨域配置完成上线。
5. 个人技术沉淀：通过博客展示技术广度与踩坑复盘能力。

因此，本方案将“面试项目”拆成 **10 篇独立可交付的小 Demo**，每篇文章都围绕某个能力点，保证：

- 文章结构统一、面试官好读好追问
- 每篇自包含：范围边界明确、本地可验证、具备排错复盘
- 每篇都体现“你能独立干完整链路”：前端/后端/部署/排错/复盘

## 约束与假设

- 博客使用 Hexo（`source/_posts` 为文章目录）。
- 文章写作风格参考现有文章：使用 `front-matter`（`title/date/tags/categories`），正文使用 `#`/`##` 标题层级与清晰分段。
- 当前实现代码不在本设计阶段编写；本文件只负责“怎么写、写什么、写到什么深度”。
- “10 篇独立小项目”在叙事上要求每篇独立，但允许在写作模板与关键设计决策上复用，以降低维护成本并保持一致性。

## 统一模板（每篇文章固定小节）

每篇文章都必须包含并按顺序覆盖以下固定小节（内容按本篇能力点替换）：

1. **前言：面试要求 -> 这篇交付了什么**
   - 用 2-3 段把招聘要求翻译成“你做出来了什么可展示结果”。
2. **本篇独立 Demo 的边界**
   - 明确：做了哪些、刻意不做哪些（避免面试官追问发散）。
3. **技术路线（架构与数据流）**
   - 8-12 行文字链路，必须出现：前端调用 -> 服务层 -> 模型/存储（按本篇实际替换）。
4. **核心实现点（只讲 2-4 个最关键难点）**
   - 每个点按“现象/问题 -> 你的方案 -> 为什么这样”写。
5. **本地测试与验证（必须可执行）**
   - 给出 `run` 的命令（或 Docker 启动命令），并写清“成功长什么样”。
6. **排错复盘：你遇到过的 3 个典型坑**
   - 每个坑按“现象 -> 根因 -> 修复 -> 如何避免回归”。
7. **面试追问准备（3-5 个问题 + 关键词答法）**
   - 每个问题都对齐本篇能力点，并给出你答题时的“关键词要点”。

## 标题/标签/命名规范（让作品集可检索）

1. 标题命名格式
   - 每篇标题开头用 ``，例如：`Streaming 打字机：React 流式 UI + 对话上下文状态`
2. Hexo front-matter 建议字段
   - `title`：包含能力点关键词
   - `date`：连续填充（从本日开始）
   - `tags`：至少包含 `面试交付`，再加对应技术点（React/Next/Streaming/Go/Rust/AI/Cursor/Docker/Nginx/TLS/CORS/复盘）
   - `categories`：建议保持一致（如 `技术教程` 或 `面试项目`）
3. 文件命名（落在 `source/_posts`）
   - 推荐：`YYYY-MM-DD-<slug>.md`，slug 由“能力点关键词”组成，保证可排序。

## 10 篇独立小项目：内容分工与验证点

> 注：下述“验证点”是文章必须落到“可观察/可复现”的要求。后续实现阶段会为每篇补齐对应 Demo 的具体命令、日志片段与最小可运行路径。

1. **React/Next 工程化骨架（独立交付）**
   - 交付范围：工程结构、页面/路由、请求封装、状态管理的基础工程化。
   - 必须验证：本地启动可用；请求层可复用（例如自定义 hook/service 层）。
2. **大模型 API：Streaming 打字机（前端流式渲染）**
   - 交付范围：从 API 到 UI 的 streaming 增量渲染（类似打字机效果）。
   - 必须验证：逐 token/逐片段增量显示；中断/重试行为可解释且可观测。
3. **复杂对话上下文：可恢复的上下文状态管理**
   - 交付范围：对话历史、角色消息、上下文裁剪策略、可恢复机制（刷新/重进策略）。
   - 必须验证：刷新/重进后上下文可恢复或有明确解释；裁剪策略能被面试官追问。
4. **AI 工具提效：Cursor/Claude 写掉 CRUD/样板代码**
   - 交付范围：用 AI 生成基础模块（DTO/Schema/路由/表单/校验等），你负责“业务正确性”与“整合”。
   - 必须验证：明确哪些模块由 AI 生成；你 review 的检查点与修复策略清晰。
5. **后端（Go 或 Rust）：完整可运行项目（干净业务逻辑拆分）**
   - 交付范围：领域模型/服务层/路由分层、可运行 API 示例、可测试性。
   - 必须验证：提供可运行 API；核心业务逻辑有清晰层次与输入输出契约。
6. **后端排错与验证：日志/错误分类/复现闭环**
   - 交付范围：面试时能讲清“怎么排错”，包含一个真实错误案例。
   - 必须验证：至少 1 个真实错误案例；说明定位链路与修复后的回归验证方式。
7. **AI 驱动的系统设计：接口契约与数据模型**
   - 交付范围：将“数据模型 + 接口契约”交给 AI 辅助，形成稳定 DTO/Schema，并解释边界条件。
   - 必须验证：接口契约（DTO/Schema）与模型约束能被你解释清楚；有“为什么这样建模”的理由。
8. **独立部署：Docker + Nginx 反代（上线可用）**
   - 交付范围：容器化、端口规划、Nginx 反代、健康检查与静态资源/前后端路由策略。
   - 必须验证：给出关键配置片段；上线后能正常访问（HTTP 层面）。
9. **证书与跨域：TLS、CORS、Streaming 兼容策略**
   - 交付范围：TLS/跨域/连接保持策略，确保 streaming 在真实域名下仍可工作。
   - 必须验证：能解释为何要这么配；至少解决过一次跨域/连接问题并复盘。
10. **踩坑复盘：本次 Demo 的 3 个最危险 bug**
   - 交付范围：真实叙事（现象 -> 根因 -> 修复 -> 防回归），聚焦最危险的 bug 类型。
   - 必须验证：每个 bug 都有“定位证据”（日志/请求/复现步骤）与“防回归动作”（测试/约束/监控）。

## 风险点与写作策略

- 风险：每篇过于泛泛，缺少“可验证细节”。
  - 策略：每篇强制包含“命令/成功标准/日志证据/面试追问关键词”。
- 风险：“AI 生成”写成口号，缺少你如何保证业务正确性。
  - 策略：每篇都要求写“AI 生成 -> 你 review -> 你修复的具体点”。
- 风险：Streaming/上下文状态管理写得像概念而非工程。
  - 策略：每篇至少 1 个真实坑点，必须描述客户端/服务端的行为差异与原因。
